왜 도메인을 분리하나

왜 3개의 별도 JWT인가

USER와 AUTHOR는 같은 사람이라도 별도 계정(Q1·Q5). 어드민은 운영자로 보안 수준이 다르다. JWT의 aud claim으로 토큰의 발급 도메인을 명시하여, 잘못된 토큰으로 다른 도메인 API 호출 시 즉시 거부 (401).

D 권한 분기 흐름 verify_delegation_permission 알고리즘

모든 콘텐츠 액션의 권한 검증

검수 제출, 예약 공개, 발행 이력 수정 등 콘텐츠 관련 액션은 모두 위임 SB H3 알고리즘으로 검증된다. POST /delegation/verify API 또는 백엔드 함수 호출로 4단계 검증을 거쳐 self / delegated / cp_master / no_permission 중 하나로 결정.

1 JWT 인증
AuthorBearer 토큰 검증. aud=author 확인. 만료·서명 검증.
2 본인이 author인가? → mode=self
SERIES.author_id == JWT.sub이면 즉시 권한 부여. 가장 빠른 경로.
3 활성 위임 존재? → mode=delegated
DELEGATION 테이블 조회 — receiver_id == JWT.sub AND issuer_id == series.author_id
AND status='active' AND valid_to > NOW() AND action ∈ action_types
AND (scope='all_series' OR series_id ∈ series_ids)
4 CP 마스터 + 정책 ON? → mode=cp_master
JWT.role == 'master' AND series.team.cp_id == JWT.cp_id
AND cp.master_uploads_without_delegation == TRUE (D-1 정책)
5 모두 실패 → 403 Forbidden · no_permission
응답: { ok: false, mode: 'no_permission', reason: 'no_permission' }
클라이언트는 [팀장에게 위임 요청] 버튼 노출 → POST /delegation/request

D1. 권한 검증 호출 예시

// 검수 제출 직전 (검수·발행 SB C4) const result = await fetch('/v1/delegation/verify', { method: 'POST', headers: { 'Authorization': 'Bearer eyJhbGc...', 'Content-Type': 'application/json' }, body: JSON.stringify({ action: 'submit_review', seriesId: 456 }) }); // 응답 { ok: true, mode: 'delegated', // 'self' | 'delegated' | 'cp_master' delegationId: 'D-2432', // delegated일 때 D-XXXX reason: null } // → 클라이언트는 mode에 따라 UI 분기: // self: 일반 검수 제출 모달 // delegated: Lavender callout (대리 업로드 안내) + X-Delegation-Id 헤더로 후속 요청 // cp_master: Maroon callout (CP 정책 모드 안내)
